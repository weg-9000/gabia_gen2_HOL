## 배경 지식

### VPC Peering이란?

VPC Peering은 두 개의 VPC를 프라이빗 네트워크로 직접 연결하는 네트워킹 기능입니다. 인터넷을 경유하지 않고 내부 IP 주소를 사용하여 통신하므로 보안성이 높고 지연 시간이 낮습니다.

```
[VPC Peering 개념도]

인터넷 경유 방식 (비권장):
+----------+     인터넷      +----------+
|  VPC-A   |  ─────────────→  |  VPC-B   |
| 10.0.0.0 |  ← 공인 IP 사용   | 10.1.0.0 |
+----------+  느림, 비용 발생  +----------+

VPC Peering 방식 (권장):
+----------+   Peering 연결   +----------+
|  VPC-A   |  ═══════════════  |  VPC-B   |
| 10.0.0.0 |  프라이빗 IP 사용  | 10.1.0.0 |
+----------+  빠름, 보안성 높음  +----------+
```

### VPC Peering의 특징

```
[핵심 특징]

1. 프라이빗 연결
   - 인터넷 경유 없음
   - 내부 IP 주소 사용
   - 트래픽 암호화 불필요 (이미 격리됨)

2. 비전이적(Non-Transitive)
   - A ↔ B, B ↔ C 연결 시
   - A → C 직접 통신 불가
   - A → C 통신 필요 시 별도 Peering 필요

3. CIDR 비중복 필수
   - 연결할 VPC의 CIDR이 겹치면 안 됨
   - 10.0.0.0/16 ↔ 10.0.0.0/16 불가
   - 10.0.0.0/16 ↔ 10.1.0.0/16 가능

4. 양방향 구성 필요
   - Peering 연결 생성
   - 양쪽 VPC 라우팅 테이블 모두 수정
   - 보안 그룹도 양쪽에서 허용 필요
```

### 네트워크 연결 방식 비교

```
[연결 방식별 비교]

방식          | VPC Peering    | VPN          | Transit Gateway
-------------|----------------|--------------|------------------
연결 유형     | VPC ↔ VPC     | VPC ↔ On-prem| 중앙 허브
확장성        | 1:1 연결       | 1:1 연결      | N:N 연결
전이성        | 비전이적       | 비전이적      | 전이적
복잡도        | 낮음          | 중간          | 높음
비용          | 낮음          | 중간          | 높음
지연시간      | 매우 낮음      | 중간          | 낮음
암호화        | 선택적         | 필수          | 선택적

권장 사용 시나리오:
- VPC Peering: 소수 VPC 간 직접 연결
- VPN: 온프레미스 데이터센터 연결
- Transit Gateway: 다수 VPC 중앙 집중 관리
```

### 비전이적(Non-Transitive) 특성 이해

```
[비전이적 라우팅]

잘못된 기대:
VPC-A ←→ VPC-B ←→ VPC-C
"A에서 B를 거쳐 C로 갈 수 있다" (X)

실제 동작:
VPC-A ←→ VPC-B    OK
VPC-B ←→ VPC-C    OK
VPC-A ←→ VPC-C    X (직접 연결 없음)

[3개 VPC 연결 시 필요한 Peering]

완전 연결 (Full Mesh):
   VPC-A
   /   \
  /     \
VPC-B ── VPC-C

필요한 Peering 연결: 3개
- A ↔ B
- B ↔ C
- A ↔ C

VPC 수에 따른 Peering 수:
- 2개 VPC: 1개 Peering
- 3개 VPC: 3개 Peering
- 4개 VPC: 6개 Peering
- N개 VPC: N*(N-1)/2 개 Peering
```

### CIDR 설계 고려사항

```
[VPC CIDR 설계 원칙]

1. 충분한 IP 공간 확보
   /16 = 65,536 IPs
   /20 = 4,096 IPs
   /24 = 256 IPs

2. 향후 확장 고려
   현재 필요: 100 IPs
   권장 설계: /24 (256 IPs) 이상

3. Peering 고려한 비중복 설계

   환경별 CIDR 할당 예시:
   +---------------+------------------+
   | 환경          | CIDR             |
   +---------------+------------------+
   | Production    | 10.0.0.0/16      |
   | Staging       | 10.1.0.0/16      |
   | Development   | 10.2.0.0/16      |
   | Shared        | 10.100.0.0/16    |
   +---------------+------------------+

   모든 환경 간 Peering 가능 (CIDR 비중복)

4. 잘못된 설계 예시
   Production: 10.0.0.0/16
   Development: 10.0.0.0/16  (중복!)
   → Peering 불가능
```

## 심화 이해

### 다중 환경 VPC 아키텍처

```
[환경별 VPC 분리 패턴]

+------------------+
|   Shared VPC     |  ← 공통 서비스
|  10.100.0.0/16   |    (모니터링, 로깅, CI/CD)
+------------------+
    |   |   |
    |   |   +--------------------+
    |   +-----------+            |
    |               |            |
+----------+  +----------+  +----------+
| Dev VPC  |  | Stg VPC  |  | Prd VPC  |
|10.2.0.0  |  |10.1.0.0  |  |10.0.0.0  |
+----------+  +----------+  +----------+

Peering 연결:
- Shared ↔ Dev
- Shared ↔ Stg
- Shared ↔ Prd
- Stg ↔ Prd (데이터 동기화용, 선택적)

이점:
- 환경 간 격리 유지
- 공통 서비스 중앙 집중화
- 필요한 연결만 선택적 구성
```

### 마이크로서비스 VPC 분리

```
[서비스별 VPC 패턴]

+-----------+     +-----------+     +-----------+
| User VPC  |←───→|  API VPC  |←───→| Data VPC  |
| 10.1.0.0  |     | 10.2.0.0  |     | 10.3.0.0  |
+-----------+     +-----------+     +-----------+
                       ↑
                       |
                  +-----------+
                  |  ML VPC   |
                  | 10.4.0.0  |
                  +-----------+

각 VPC 역할:
- User VPC: 프론트엔드, 사용자 인터페이스
- API VPC: API 게이트웨이, 비즈니스 로직
- Data VPC: 데이터베이스, 캐시
- ML VPC: 머신러닝 추론 서비스

장점:
- 서비스별 독립 배포
- 장애 격리 (Blast Radius 최소화)
- 팀별 독립 관리
- 보안 경계 명확화
```

### 하이브리드 클라우드 연결

```
[온프레미스 + 클라우드 연결]

+-------------------+
| On-Premises DC    |
| 192.168.0.0/16    |
+-------------------+
         |
         | VPN 연결
         |
+-------------------+
|  Transit VPC      |  ← 연결 중심지
|  10.255.0.0/16    |
+-------------------+
    |       |
    |       +------------------+
    |                          |
+----------+            +----------+
| Prod VPC |            | Dev VPC  |
|10.0.0.0  |            |10.1.0.0  |
+----------+            +----------+

라우팅 구성:
- On-Prem → Transit VPC: VPN 터널
- Transit VPC → Prod/Dev: VPC Peering
- Prod/Dev → On-Prem: Transit VPC 경유

주의: Transit VPC를 통한 전이적 라우팅은
     추가 설정 필요 (NAT/프록시)
```

### 보안 강화 패턴

```
[계층별 보안 적용]

1. VPC 수준 격리
   - 환경별 VPC 분리
   - CIDR 범위로 접근 제어
   - 필요한 Peering만 생성

2. 서브넷 수준
   - 퍼블릭/프라이빗 분리
   - 라우팅으로 접근 경로 제한

3. 보안 그룹 수준
   - Peering 대상 CIDR만 허용
   - 필요한 포트만 개방
   - 보안 그룹 참조 활용 (동일 VPC 내)

4. 애플리케이션 수준
   - mTLS 적용
   - API 인증
   - 감사 로깅

[최소 권한 Peering 규칙 예시]

app-sg (shop-vpc):
- 8000/TCP ← 10.1.1.0/24    (dev-subnet만)
- 8000/TCP ← 10.1.2.0/24    (stg-subnet만)
- 다른 서브넷은 차단

전체 VPC 허용 vs 특정 서브넷 허용:
- 10.1.0.0/16: dev-vpc 전체 허용
- 10.1.1.0/24: dev-subnet만 허용 (권장)
```

### Peering 제한사항과 대안

```
[VPC Peering 제한사항]

1. CIDR 중복 불가
   문제: 인수합병 시 기존 CIDR 충돌
   대안: NAT 게이트웨이로 IP 변환

2. 비전이적 라우팅
   문제: 다수 VPC 연결 시 복잡도 급증
   대안: Transit Gateway 사용

3. 대역폭 제한
   문제: 고대역폭 요구 시
   대안: Direct Connect, 전용선

4. 리전 간 지연
   문제: 다른 리전 간 Peering 시 지연
   대안: 같은 리전 우선, 필요시 캐싱

[VPC 수 증가에 따른 복잡도]

VPC 수  | Peering 수 | 관리 복잡도
--------|-----------|------------
2       | 1         | 낮음
3       | 3         | 낮음
5       | 10        | 중간
10      | 45        | 높음
20      | 190       | 매우 높음

권장:
- 5개 이하: VPC Peering
- 5개 초과: Transit Gateway 검토
```

### 비용 고려사항

```
[VPC Peering 비용 구조]

1. Peering 연결 자체
   - 연결 생성: 무료
   - 연결 유지: 무료

2. 데이터 전송
   - 같은 가용 영역: 무료 또는 저렴
   - 다른 가용 영역: 영역 간 요금 적용
   - 다른 리전: 리전 간 요금 적용

[비용 최적화 전략]

1. 가용 영역 정렬
   VPC-A: AZ-a    ←→    VPC-B: AZ-a
   (같은 AZ끼리 통신 시 비용 절감)

2. 트래픽 최소화
   - 필요한 데이터만 전송
   - 압축 적용
   - 캐싱 활용

3. 불필요한 Peering 제거
   - 미사용 Peering 정기 점검
   - 트래픽 모니터링
```

## 트러블슈팅

### 문제 1: Peering 연결 상태가 "pending"

```
[증상]
Peering 연결 생성 후 상태가 계속 "pending-acceptance"

[원인]
수락자(Accepter) VPC에서 수락하지 않음

[해결 방법]

1. 같은 계정 내 Peering
   콘솔 > VPC > VPC Peering > 요청 수락 클릭

2. 다른 계정 간 Peering
   수락자 계정에서:
   - 콘솔 로그인
   - VPC > VPC Peering > 수신된 요청 확인
   - 요청 수락

3. 수락 대기 시간
   - 7일 내 수락 필요
   - 미수락 시 자동 만료
```

### 문제 2: Peering 연결됐지만 Ping 실패

```
[증상]
Peering 상태는 "active"지만 ping 응답 없음

[원인 분석 순서]

1. 라우팅 테이블 확인
   양쪽 VPC 모두 상대방 CIDR 라우트 있는지

   shop-vpc 라우팅:
   10.1.0.0/16 → pcx-xxx  [있어야 함]

   dev-vpc 라우팅:
   10.0.0.0/16 → pcx-xxx  [있어야 함]

2. 보안 그룹 확인
   ICMP(ping) 허용 여부

   app-sg:
   ICMP ← 10.1.0.0/16  [있어야 함]

   dev-sg:
   ICMP ← 10.0.0.0/16  [있어야 함]

3. 네트워크 ACL 확인
   서브넷의 NACL에서 차단 여부

4. OS 방화벽 확인
   # Linux
   iptables -L -n | grep icmp

[진단 명령어]
# 라우팅 확인
[인스턴스]$ ip route

# 연결 추적
[인스턴스]$ traceroute 10.x.x.x

# 패킷 캡처
[인스턴스]$ tcpdump -i eth0 icmp
```

### 문제 3: CIDR 중복으로 Peering 실패

```
[증상]
"CIDR block overlaps" 오류

[원인]
두 VPC의 CIDR이 겹침

예:
shop-vpc: 10.0.0.0/16
legacy-vpc: 10.0.0.0/16  (중복!)

[해결 방법]

옵션 1: VPC CIDR 변경 (권장하지 않음)
- 기존 VPC CIDR 변경 불가
- 새 VPC 생성 필요
- 리소스 마이그레이션 필요

옵션 2: 보조 CIDR 추가
- VPC에 비중복 보조 CIDR 추가
- 새 서브넷을 보조 CIDR에 생성
- 보조 CIDR로 Peering
- 제한: 일부 클라우드에서 미지원

옵션 3: NAT 기반 연결
- NAT를 통한 IP 변환
- 복잡하지만 CIDR 변경 불필요

옵션 4: PrivateLink 사용
- 서비스 단위 연결
- CIDR 무관
- 특정 서비스만 노출
```

### 문제 4: 일부 서브넷만 통신 안 됨

```
[증상]
같은 VPC 내 일부 서브넷만 Peering 통신 불가

[원인]
서브넷별로 다른 라우팅 테이블 사용

[확인 방법]

1. 서브넷 라우팅 테이블 확인
   콘솔 > VPC > 서브넷 > [서브넷 선택] > 라우팅 테이블

   public-subnet: main-rt (Peering 라우트 있음)
   private-subnet: private-rt (Peering 라우트 없음!)

2. 누락된 라우팅 테이블에 라우트 추가
   private-rt에도 Peering 라우트 추가

[해결 방법]
모든 관련 라우팅 테이블에 Peering 라우트 추가

main-rt:
- 10.1.0.0/16 → pcx-xxx

private-rt:
- 10.1.0.0/16 → pcx-xxx  [추가 필요]
```

### 문제 5: 비대칭 라우팅

```
[증상]
한 방향은 통신되지만 반대 방향 응답 없음

[원인]
요청과 응답이 다른 경로로 전달

예:
요청: A → B (Peering 경유)
응답: B → A (인터넷 경유) → 실패

[진단]
# 양방향 traceroute
[A]$ traceroute 10.1.1.10
[B]$ traceroute 10.0.2.10

# 경로가 다르면 비대칭

[해결 방법]

1. 라우팅 테이블 재확인
   양쪽 모두 Peering 경유하도록 설정

2. 보안 그룹 검토
   Stateful이므로 응답은 자동 허용되어야 함
   그러나 경로가 다르면 다른 연결로 인식

3. 더 구체적인 라우트 사용
   0.0.0.0/0보다 10.x.0.0/16이 우선
   명시적 라우트로 경로 고정
```
