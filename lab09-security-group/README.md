## 배경 지식

### 보안 그룹이란?

보안 그룹(Security Group)은 클라우드 환경에서 인스턴스 수준의 가상 방화벽입니다. 네트워크 트래픽의 인바운드(들어오는)와 아웃바운드(나가는) 규칙을 정의하여 허용되지 않은 접근을 차단합니다.

```
[보안 그룹의 특징]

1. 상태 저장(Stateful) 방화벽
   - 인바운드로 허용된 연결의 응답은 자동 허용
   - 별도의 아웃바운드 규칙 없이도 응답 가능

   예시:
   클라이언트 ----[요청: 80 포트]----> 웹서버
   클라이언트 <---[응답: 자동 허용]---- 웹서버

2. 허용 규칙만 정의
   - 명시적 거부(Deny) 규칙 없음
   - 규칙에 없는 트래픽은 자동 거부

3. 인스턴스 수준 적용
   - 각 인스턴스에 하나 이상의 보안 그룹 연결
   - 동일 서브넷 내에서도 인스턴스마다 다른 정책 가능
```

### 보안 그룹 vs 네트워크 ACL 비교

```
[비교표]

항목              | 보안 그룹           | 네트워크 ACL
-----------------|--------------------|-----------------
적용 수준         | 인스턴스            | 서브넷
상태             | Stateful           | Stateless
규칙 유형         | 허용만              | 허용/거부
규칙 평가         | 모든 규칙 평가       | 순서대로 평가
기본 정책         | 모두 거부           | 모두 허용
연결 추적         | 있음               | 없음

권장 사용 패턴:
- 보안 그룹: 애플리케이션별 세부 제어
- 네트워크 ACL: 서브넷 전체 광범위 제어
```

### Stateful vs Stateless 이해

```
[Stateful 방화벽 - 보안 그룹]

1. 인바운드 규칙: TCP 80 허용
2. 클라이언트가 80 포트로 요청
3. 서버가 응답 (아웃바운드)
4. 응답은 자동 허용됨 (연결 추적)

요청:  클라이언트:45678 → 서버:80     [인바운드 규칙 검사]
응답:  서버:80 → 클라이언트:45678     [자동 허용]

[Stateless 방화벽 - 네트워크 ACL]

1. 인바운드 규칙: TCP 80 허용
2. 아웃바운드 규칙: 임시 포트(1024-65535) 허용 필요
3. 각 방향마다 별도 규칙 필수

요청:  클라이언트:45678 → 서버:80     [인바운드 규칙 검사]
응답:  서버:80 → 클라이언트:45678     [아웃바운드 규칙 검사]
```

### 포트 번호와 프로토콜 기초

```
[주요 포트 번호]

Well-Known Ports (0-1023):
- 22/TCP    : SSH (보안 원격 접속)
- 80/TCP    : HTTP (웹 서버)
- 443/TCP   : HTTPS (보안 웹 서버)
- 3306/TCP  : MySQL
- 5432/TCP  : PostgreSQL
- 6379/TCP  : Redis
- 27017/TCP : MongoDB

Registered Ports (1024-49151):
- 3000/TCP  : Node.js 개발 서버 (관례)
- 8000/TCP  : Python/Django 개발 서버
- 8080/TCP  : 대체 HTTP 포트
- 8443/TCP  : 대체 HTTPS 포트

Dynamic Ports (49152-65535):
- 클라이언트 임시 포트 (ephemeral ports)
- OS가 자동 할당
```

### 최소 권한 원칙 (Principle of Least Privilege)

```
[최소 권한 원칙이란?]

정의: 시스템의 모든 구성 요소가 작업 수행에 필요한
     최소한의 권한만 가져야 한다는 보안 원칙

[잘못된 예]
보안 그룹: all-open-sg
- Inbound: 0.0.0.0/0 → 모든 포트
- 결과: 모든 곳에서 모든 서비스 접근 가능
- 위험: 브루트포스, 취약점 스캔, 무단 접근

[올바른 예]
보안 그룹: web-sg
- Inbound:
  - 80/TCP  ← 0.0.0.0/0    (웹 서비스)
  - 443/TCP ← 0.0.0.0/0    (보안 웹 서비스)
  - 22/TCP  ← 10.0.0.5/32  (관리자 IP만)
- 결과: 필요한 서비스만 필요한 곳에서 접근
- 효과: 공격 표면(Attack Surface) 최소화
```

### 보안 그룹 참조 (Security Group Reference)

```
[IP 기반 규칙의 한계]

상황: 앱 서버에서 DB 접근 허용
방법 1 - IP 직접 지정:
  db-sg: 5432/TCP ← 10.0.2.10/32

문제점:
- 앱 서버 IP 변경 시 규칙 수정 필요
- 앱 서버 추가 시 규칙 추가 필요
- 오토스케일링 시 관리 불가능

[보안 그룹 참조의 장점]

방법 2 - 보안 그룹 참조:
  db-sg: 5432/TCP ← app-sg

장점:
- app-sg가 적용된 모든 인스턴스 자동 허용
- 인스턴스 IP 변경 무관
- 오토스케일링 완벽 지원
- 규칙 관리 단순화

동작 원리:
  app-sg에 속한 인스턴스
      ↓
  소스 IP 확인 (10.0.2.10)
      ↓
  해당 IP가 app-sg에 속함 확인
      ↓
  db-sg 규칙에서 app-sg 허용 확인
      ↓
  접근 허용
```

## 실습 단계

### 1단계: 보안 그룹 설계

실습 환경의 아키텍처를 기반으로 필요한 보안 그룹을 설계합니다.

```
[3-Tier 아키텍처 보안 설계]

인터넷
   |
   ↓ [80, 443]
+-----------------+
|    web-sg       |  ← 로드밸런서/웹서버
+-----------------+
   |
   ↓ [8000]
+-----------------+
|    app-sg       |  ← 애플리케이션 서버
+-----------------+
   |
   ↓ [5432]
+-----------------+
|    db-sg        |  ← 데이터베이스 서버
+-----------------+

추가 보안 그룹:
+-----------------+
|  bastion-sg     |  ← 관리용 점프 서버
+-----------------+
```

### 2단계: Bastion(점프 서버) 보안 그룹 생성

관리 목적의 SSH 접속을 위한 보안 그룹을 먼저 생성합니다.

```
[콘솔 경로]
네트워킹 > 보안 그룹 > 보안 그룹 생성

[설정 값]
기본 정보:
- 이름: bastion-sg
- 설명: 점프 서버용 보안 그룹
- VPC: shop-vpc

Inbound 규칙:
+------+----------+------------------+------------------+
| 유형  | 프로토콜  | 포트             | 소스             |
+------+----------+------------------+------------------+
| SSH  | TCP      | 22               | [관리자 IP]/32   |
+------+----------+------------------+------------------+

소스 IP 확인 방법:
- 브라우저에서 "what is my ip" 검색
- 또는 터미널: curl ifconfig.me

예시: 203.0.113.50/32

Outbound 규칙:
+------+----------+------------------+------------------+
| 유형  | 프로토콜  | 포트             | 대상             |
+------+----------+------------------+------------------+
| All  | All      | All              | 0.0.0.0/0        |
+------+----------+------------------+------------------+
```

### 3단계: 웹 서버 보안 그룹 생성

외부 트래픽을 받는 웹 서버/로드밸런서용 보안 그룹입니다.

```
[설정 값]
기본 정보:
- 이름: web-sg
- 설명: 웹 서버 및 로드밸런서용
- VPC: shop-vpc

Inbound 규칙:
+-------+----------+------+-------------------+-------------------+
| 유형   | 프로토콜  | 포트  | 소스              | 설명              |
+-------+----------+------+-------------------+-------------------+
| HTTP  | TCP      | 80   | 0.0.0.0/0         | 웹 트래픽         |
| HTTPS | TCP      | 443  | 0.0.0.0/0         | 보안 웹 트래픽     |
| SSH   | TCP      | 22   | bastion-sg        | 관리 접속         |
+-------+----------+------+-------------------+-------------------+

주의사항:
- HTTP/HTTPS는 전체 공개 (0.0.0.0/0)
- SSH는 Bastion에서만 허용 (보안 그룹 참조)
- 직접 SSH 접속 차단으로 보안 강화

Outbound 규칙:
+------+----------+------+-------------+
| 유형  | 프로토콜  | 포트  | 대상        |
+------+----------+------+-------------+
| All  | All      | All  | 0.0.0.0/0   |
+------+----------+------+-------------+
```

### 4단계: 애플리케이션 서버 보안 그룹 생성

웹 서버에서만 접근 가능한 애플리케이션 서버용 보안 그룹입니다.

```
[설정 값]
기본 정보:
- 이름: app-sg
- 설명: 애플리케이션 서버용 (FastAPI)
- VPC: shop-vpc

Inbound 규칙:
+---------+----------+------+-------------+-------------------+
| 유형     | 프로토콜  | 포트  | 소스        | 설명              |
+---------+----------+------+-------------+-------------------+
| Custom  | TCP      | 8000 | web-sg      | FastAPI 트래픽    |
| SSH     | TCP      | 22   | bastion-sg  | 관리 접속         |
+---------+----------+------+-------------+-------------------+

핵심 보안 원칙:
- 8000 포트는 web-sg에서만 접근 가능
- 외부에서 직접 앱 서버 접근 불가
- SSH는 Bastion 경유 필수

Outbound 규칙:
+------+----------+------+-------------+
| 유형  | 프로토콜  | 포트  | 대상        |
+------+----------+------+-------------+
| All  | All      | All  | 0.0.0.0/0   |
+------+----------+------+-------------+

아웃바운드 전체 허용 이유:
- 외부 API 호출
- 패키지 다운로드 (pip, apt 등)
- DNS 조회
```

### 5단계: 데이터베이스 보안 그룹 생성

가장 민감한 데이터베이스를 보호하는 보안 그룹입니다.

```
[설정 값]
기본 정보:
- 이름: db-sg
- 설명: PostgreSQL 데이터베이스용
- VPC: shop-vpc

Inbound 규칙:
+------------+----------+------+-------------+-------------------+
| 유형        | 프로토콜  | 포트  | 소스        | 설명              |
+------------+----------+------+-------------+-------------------+
| PostgreSQL | TCP      | 5432 | app-sg      | DB 연결           |
| SSH        | TCP      | 22   | bastion-sg  | 관리 접속         |
+------------+----------+------+-------------+-------------------+

보안 강화:
- DB 포트는 app-sg에서만 접근
- 외부 직접 접근 완전 차단
- 웹 서버에서도 직접 DB 접근 불가

Outbound 규칙:
+--------+----------+---------+-------------+
| 유형    | 프로토콜  | 포트     | 대상        |
+--------+----------+---------+-------------+
| HTTPS  | TCP      | 443     | 0.0.0.0/0   |
+--------+----------+---------+-------------+

아웃바운드 제한 이유:
- DB는 외부 통신 최소화
- 보안 업데이트용 HTTPS만 허용
- 데이터 유출 방지
```

### 6단계: 보안 그룹 적용

생성한 보안 그룹을 인스턴스에 적용합니다.

```
[인스턴스에 보안 그룹 적용]

콘솔 경로:
컴퓨팅 > 가상 서버 > [인스턴스 선택] > 보안 그룹 변경

적용 예시:
- bastion-server   → bastion-sg
- web-server       → web-sg
- app-server       → app-sg
- database-server  → db-sg

[다중 보안 그룹 적용]

하나의 인스턴스에 여러 보안 그룹 적용 가능:
- monitoring-server → app-sg + monitoring-sg

규칙 병합:
- 여러 보안 그룹의 규칙이 합쳐짐 (Union)
- 하나라도 허용하면 트래픽 통과
```

### 7단계: 보안 그룹 테스트

설정한 보안 규칙이 정상 동작하는지 검증합니다.

```
[테스트 시나리오]

1. Bastion 접속 테스트
   로컬 PC → Bastion (SSH)

   # 관리자 IP에서 접속
   ssh -i key.pem user@bastion-public-ip
   결과: 성공 (관리자 IP 허용)

   # 다른 IP에서 접속 시도
   결과: 타임아웃 (허용되지 않음)

2. 웹 서버 접속 테스트
   # HTTP 접속
   curl http://web-server-ip
   결과: 성공 (80 포트 전체 허용)

   # 직접 SSH 시도 (Bastion 경유 없이)
   ssh user@web-server-ip
   결과: 타임아웃 (직접 SSH 차단)

3. 앱 서버 접속 테스트
   # 외부에서 직접 접속
   curl http://app-server-ip:8000
   결과: 타임아웃 (외부 접근 차단)

   # 웹 서버에서 접속
   [web-server]$ curl http://app-server-private-ip:8000
   결과: 성공 (web-sg에서 허용)

4. DB 접속 테스트
   # 앱 서버에서 접속
   [app-server]$ psql -h db-private-ip -U postgres
   결과: 성공 (app-sg에서 허용)

   # 웹 서버에서 직접 접속
   [web-server]$ psql -h db-private-ip -U postgres
   결과: 거부 (web-sg 허용 안 됨)
```

## 심화 이해

### 계층별 방어 (Defense in Depth)

```
[다중 보안 계층 구성]

Layer 1: 네트워크 경계
+----------------------------------+
|  인터넷 게이트웨이                 |
|  - DDoS 방어                      |
|  - 기본 필터링                     |
+----------------------------------+
              |
              ↓
Layer 2: 서브넷 수준
+----------------------------------+
|  네트워크 ACL                     |
|  - 서브넷 전체 광범위 제어          |
|  - IP 블록 차단                   |
+----------------------------------+
              |
              ↓
Layer 3: 인스턴스 수준
+----------------------------------+
|  보안 그룹                        |
|  - 인스턴스별 세부 제어            |
|  - 포트/프로토콜 제어              |
+----------------------------------+
              |
              ↓
Layer 4: 애플리케이션 수준
+----------------------------------+
|  WAF (Web Application Firewall)  |
|  - SQL Injection 차단            |
|  - XSS 공격 차단                  |
+----------------------------------+
              |
              ↓
Layer 5: 호스트 수준
+----------------------------------+
|  OS 방화벽 (iptables/firewalld)  |
|  - 최종 방어선                    |
|  - 호스트 기반 규칙               |
+----------------------------------+
```

### 마이크로서비스 환경 보안 그룹

```
[서비스별 보안 그룹 분리]

+-------------+     +-------------+     +-------------+
|   api-sg    |     |   auth-sg   |     |  payment-sg |
| Port: 8080  |     | Port: 8081  |     | Port: 8082  |
+-------------+     +-------------+     +-------------+
      ↓                   ↓                   ↓
+-------------+     +-------------+     +-------------+
| user-db-sg  |     | auth-db-sg  |     | payment-db  |
| Port: 5432  |     | Port: 5432  |     | Port: 5432  |
+-------------+     +-------------+     +-------------+

서비스 간 통신 규칙:
- api-sg → auth-sg:8081 (인증 요청)
- api-sg → payment-sg:8082 (결제 요청)
- auth-sg는 자신의 DB만 접근
- payment-sg는 자신의 DB만 접근

장점:
- 서비스 격리로 보안 강화
- 침해 시 영향 범위 최소화
- 서비스별 독립적 보안 정책
```

스케일링 그룹 설정:

DB 보안 그룹:
- 소스: app-sg (IP 아닌 보안 그룹)
- 결과: 모든 앱 인스턴스 자동 허용

동작 흐름:
1. 트래픽 증가 → 앱 서버 추가
2. 새 인스턴스에 app-sg 자동 적용
3. DB는 app-sg 참조로 자동 허용
4. 트래픽 감소 → 앱 서버 제거
5. 별도 규칙 수정 불필요

mesh-sg:

### 문제 1: 연결 타임아웃

```
[증상]
curl이나 ssh 명령이 응답 없이 멈춤

[원인 분석]

1. 보안 그룹 규칙 확인
   콘솔 > 보안 그룹 > [해당 SG] > Inbound 규칙

   확인 사항:
   - 포트 번호 정확한가?
   - 프로토콜 맞는가? (TCP/UDP)
   - 소스 IP/CIDR 정확한가?

2. 소스 IP 확인
   # 내 공인 IP 확인
   curl ifconfig.me

   # 보안 그룹에 등록된 IP와 비교
   규칙: 203.0.113.50/32
   현재 IP: 203.0.113.51  → 불일치!

3. CIDR 범위 확인
   /32 : 단일 IP (203.0.113.50만)
   /24 : 256개 IP (203.0.113.0 ~ 203.0.113.255)
   /16 : 65,536개 IP (203.0.0.0 ~ 203.0.255.255)

[해결 방법]
- 정확한 IP로 규칙 수정
- 또는 CIDR 범위 확대 (보안 주의)
```

### 문제 2: 연결 거부 (Connection Refused)

```
[증상]
"Connection refused" 오류 발생

[원인 분석]

이 오류는 보안 그룹이 아닌 서버 문제!
- 보안 그룹 차단: 타임아웃
- 서버 포트 미오픈: Connection refused

1. 서버에서 포트 확인
   # 리스닝 포트 확인
   netstat -tlnp | grep 8000
   ss -tlnp | grep 8000

   출력 없음 → 애플리케이션 미실행

2. 애플리케이션 상태 확인
   # 프로세스 확인
   ps aux | grep python
   ps aux | grep uvicorn

   # 서비스 상태
   systemctl status myapp

3. 애플리케이션 바인딩 주소 확인
   잘못된 예: 127.0.0.1:8000 (로컬만)
   올바른 예: 0.0.0.0:8000 (모든 인터페이스)

[해결 방법]
- 애플리케이션 시작
- 올바른 IP에 바인딩
```

   형식: sg-xxxxxxxxx

- 보안 그룹 ID 재확인
